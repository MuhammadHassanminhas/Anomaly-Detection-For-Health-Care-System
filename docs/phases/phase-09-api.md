# Phase 9 — API

## Objective

FastAPI backend exposing the system to humans and to the Phase 10 UI: findings queue, lifecycle actions, explanations, check management, review gate, and run/calibration visibility — practice-scoped (D-008), with a pluggable auth layer (real roles, stubbed identity provider per D-007) and the full pipeline loop proven end-to-end through the real API.

## Preconditions

- Phases 3–8 closed: every operation the API exposes already exists as a tested service; this phase adds the HTTP surface, not new business logic (any gap found ⇒ it lands in the owning module with its own tests, not in a route handler).
- D-007 stub approach confirmed (already the DECISIONS.md recommendation; real IdP wiring is Phase 11).

## Steps (one deliverable each; TDD — contract tests before handlers)

1. **App skeleton + auth layer.** FastAPI app factory; auth middleware with a provider interface (`StubProvider` now: identities+roles from signed dev tokens; OIDC provider lands in Phase 11); role model live: `triage_user` (queue, lifecycle actions), `check_reviewer` (+ review gate), `admin` (+ check/config management). Practice scoping enforced centrally: every data route requires practice context; a user's permitted-practices claim filters every query (D-008).
   *Deliverable:* auth + scoping tests: no token ⇒ 401; wrong role ⇒ 403; practice A token never sees practice B data (test-proven on every data route via a shared fixture).
2. **Findings queue + detail.** `GET /practices/{id}/findings` (filter: status, category, severity, check; sort; keyset pagination); `GET /findings/{id}` (evidence, narrative + validation status, lifecycle history from `finding_events`, check rationale + resolution guidance).
   *Deliverable:* contract tests incl. pagination stability and filter combinations against seeded fixture findings.
3. **Lifecycle actions.** `POST /findings/{id}/acknowledge|dismiss|snooze|reopen` — thin delegates to the Phase 6 feedback service; dismiss without reason code ⇒ 422 with the reason-code enum in the error; every action records the authenticated actor.
   *Deliverable:* tests per action incl. the 422 path and actor attribution; idempotency/conflict handling (double-dismiss ⇒ 409).
4. **Check management + review gate.** `GET /checks` (+ versions, per-practice config/precision/demotion state); review endpoints mirroring the Phase 4 CLI: list drafts, draft detail (definition, rationale, compiled SQL, dry-run result), approve/reject/amend — same service layer as the CLI, so the no-bypass guarantee holds (one code path to `active`); `check_reviewer` role required.
   *Deliverable:* review-flow contract tests; bypass test extended to cover the API (no route writes status outside the review service).
5. **Runs, calibration & discovery visibility.** `GET /runs` (+ per-run report data: costs, tri-state counts, narration stats); `GET /practices/{id}/calibration` (params, learned vs manual, shift history); `GET /discovery/signals` (admin-only — internal queue stays internal; served for the Phase 10 admin screens, never for triage users).
   *Deliverable:* read-only contract tests + role tests (triage_user gets 403 on discovery).
6. **Operational hardening (API-level).** Uniform error envelope; request-id middleware tied into structlog; OpenAPI schema exported as a build artifact (`artifacts/openapi.json` — the Phase 10 UI's typed contract); rate limiting deferred to Phase 11 (noted, not forgotten).
   *Deliverable:* OpenAPI artifact generated by `scripts/export_openapi.ps1`; error-envelope tests.
7. **End-to-end loop through the real API.** One test: seeded anomaly in fixture DB → pipeline run → finding appears in the queue (API) → detail shows validated narrative (API) → dismissal with reason code (API) → precision job → precision stats visible (API) → below-floor scenario ⇒ demotion visible (API).
   *Deliverable:* `tests/e2e/test_api_loop.py` green against a real server process (not TestClient only) with fixture DBs.

## Exit criteria

1. `scripts/check.ps1` exits 0, incl. all contract tests and the API e2e loop (step 7).
2. Practice isolation proven on every data route (shared cross-tenant test fixture, criterion enforced by a route-coverage assertion — a new unlisted route fails CI until it has the isolation test).
3. Role enforcement proven per route class; review-gate no-bypass test covers API + CLI.
4. Dismissal without reason code rejected at API (422), service, and DB layers (all three demonstrated).
5. `artifacts/openapi.json` exported and committed; server boots from a clean checkout via `scripts/serve.ps1` with only documented env vars.
6. Discovery endpoints inaccessible to `triage_user` (F9 stays internal at the HTTP layer too).

## Risks / dependencies / open questions

- **Auth stub must not leak into production:** the stub provider refuses to load unless `CDSS_ENV=dev|test` (test-enforced); Phase 11 wires the real provider and the production build rejects stub config.
- **Queue query performance** at realistic finding volumes: keyset pagination + app-DB indexes from Phase 3; measured properly in Phase 11's load test (D-011 p95 target) — API is built to those access patterns now.
- **Contract drift with the UI:** `openapi.json` is the single typed contract; Phase 10 generates its client from it, so drift is a build failure, not a runtime surprise.
- **Depends on:** Phases 3–8 services; D-007 (stub confirmed), D-008 (practice scoping).
- No new `DECISIONS.md` entries required by this spec.
